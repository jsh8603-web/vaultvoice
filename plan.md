# VaultVoice v3 — AI 노트 허브 리디자인 계획

## 핵심 철학
> **어떤 asset이든 텍스트와 쌍이 되어(변환) 저장되어야 한다.**
> 99_vaultvoice는 임시 수신함. Obsidian에서 태깅 → 주제별 폴더로 이동.

## 현재 수정 계획 (v3.1 패치)

### 요청 3가지

| # | 요청 | 현재 상태 | 수정 내용 |
|---|------|----------|----------|
| 1 | **피드 카드 클릭 → 상세 보기** | 카드가 클릭 불가, 미리보기만 표시 | 카드 클릭 시 노트 전문을 오버레이로 표시 |
| 2 | **상세 오버레이 닫기 UX 개선** | 닫기 버튼이 우측 상단에만 있어서 긴 글 스크롤 후 다시 위로 올려야 함 | 하단 고정 닫기 바 추가 + 빈 영역 탭으로도 닫기 |
| 3 | **음성 전사 후 문장 다듬기(리파인)** | Gemini 전사 원문을 그대로 저장 | 전사 후 Gemini 2차 호출로 문맥 기반 문장 정리 + 정리 전/후 모두 보관 |

### 수정 1: 피드 카드 클릭 → 상세 보기

**파일**: `public/app.js`, `public/index.html`

**현재 문제**:
- `renderFeedCards()`가 카드 HTML만 생성, 클릭 이벤트 없음
- `openPreview(date)`는 날짜 기반으로 레거시 `/api/daily/:date`를 호출 — 개별 노트 열기 불가

**수정 계획**:
1. `renderFeedCards()`에서 각 카드에 `data-filename` 속성 추가 (노트 파일명)
2. 카드 클릭 시 `openNoteDetail(filename)` 호출
3. 새 API 엔드포인트 `GET /api/note/:filename` 추가 — 개별 노트 전문 반환
4. `#hist-overlay`를 공용 노트 상세 오버레이로 재사용
5. server.js에 `/api/note/:filename` 엔드포인트 추가 (99_vaultvoice/ 내 파일 읽기)

**API 스펙**:
```
GET /api/note/:filename
→ { filename, frontmatter: {...}, body: "..." }
```

### 수정 2: 피드 카드 상세 → 스와이프 카드 뷰

**파일**: `public/app.js`, `public/app.css`, `public/index.html`

**현재 문제**:
- 카드 상세를 열면 닫기 버튼이 우측 상단에만 있어 긴 글 스크롤 후 다시 올라가야 함
- 카드 간 이동이 불가능

**수정 계획**:
1. 상세 보기를 카드 스타일 오버레이로 표시
2. **왼쪽 스와이프** → 닫기 (카드가 왼쪽으로 사라지며 닫힘)
3. **오른쪽 스와이프** → 같은 날짜의 다음 카드로 이동
4. 배경 탭으로도 닫기 가능
5. 터치 제스처: `touchstart/touchmove/touchend` 이벤트 처리
6. 카드 인디케이터 (현재 n/총m) 표시

---

### 수정 3: 음성 전사 리파인 (문장 다듬기)

**파일**: `server.js`

**현재 상태**:
- Step 4에서 `**화자1**: 원문 그대로` 형식으로 저장
- 구어체 반복, 어미 끊김, "어~", "그~" 등 필러가 포함됨

**수정 계획 — Step 3.5 추가 (전사 후 리파인)** (리서치 결과 반영):

**리서치 핵심 발견**:
- LLM End-to-End 정제가 현재 표준 → Gemini 2차 호출 방식 적합
- JSON responseSchema 사용이 품질/일관성 우수 → 기존 패턴 유지
- 화자별 개별 처리 + 이전 컨텍스트가 베스트 프랙티스
- 회의록: "~습니다"체 통일 / 개인 메모: 원문 톤 유지
- Obsidian: 정리본 메인 + `<details>` 원문 접기가 최적 UX

**구현**:
1. 전사 완료 후 Gemini 2차 호출 — 역할 부여 + 단계별 지침 프롬프트
2. 프롬프트 구성:
   - **역할**: "전문 회의록/메모 정리가" 부여
   - **단계별 지침**: 필러 제거 → 구어체 변환 → 문장 재구성 → 구두점 교정
   - **제약 조건**: 원본 의미 변경 금지, 정보 추가 금지, 화자 정보 유지
   - **톤 규칙**: 회의(meeting) → "~습니다"체 / 음성메모(voice) → 원문 톤 유지
3. JSON responseSchema로 구조화 출력:
   ```json
   { "refined": [{ "speaker": "화자1", "text": "정리된 문장" }] }
   ```
4. 노트 저장 — `<details>` 접기 패턴:
   ```
   ## 요약
   ...
   ## 전사 (정리)
   리파인된 전사 내용

   <details><summary>원본 전사 확인</summary>

   원본 전사 내용

   </details>
   ```
5. frontmatter에 `리파인: true/false` 필드 추가
6. 리파인 실패 시 원문만 저장 (fallback) — 최대 2회 재시도

**리파인 프롬프트**:
```
# 역할
당신은 전문적인 회의록/메모 정리가입니다.

# 목표
구어체로 전사된 내용을 읽기 좋은 문어체로 정제합니다.

# 단계별 처리 지침
1. 필러("어", "음", "그", "이제", "그러니까") 및 군말 제거
2. 반복된 어구 통합 ("회의를... 그러니까 회의를 진행" → "회의를 진행")
3. 끊어진 문장을 문법에 맞게 자연스럽게 연결
4. 문맥에 맞는 구두점(쉼표, 마침표) 추가
5. {tone_rule}

# 제약 조건
- 원문에 없는 새로운 정보를 추가 금지
- 발언의 핵심 의미나 뉘앙스 왜곡 금지
- 화자 정보(화자1, 화자2 등) 변경 금지

# 출력
JSON: { "refined": [{ "speaker": "화자1", "text": "정리된 문장" }, ...] }
```
tone_rule: meeting → "존댓말 '~습니다/ㅂ니다'체로 통일" / voice → "원문의 말투와 톤을 유지하되 필러만 제거"

### 구현 순서

```
1. server.js — GET /api/note/:filename 엔드포인트 추가
2. server.js — 음성 전사 리파인 Step 3.5 추가 (리서치 반영)
3. app.js — 피드 카드 클릭 → openNoteDetail() + 스와이프 제스처
4. app.css — 스와이프 카드 오버레이 스타일 + 애니메이션
5. 테스트 + 배포
```

### 파일 변경 목록

| 파일 | 변경 내용 |
|------|----------|
| `server.js` | `/api/note/:filename` 추가, 전사 리파인 Step 3.5 (역할부여+단계별 프롬프트) |
| `public/app.js` | `openNoteDetail()`, 피드 카드 클릭, 스와이프 제스처 (좌=닫기, 우=다음) |
| `public/app.css` | `.note-detail-overlay` 카드 스타일 + 스와이프 애니메이션 |
